Denicek: Computational substrate for document-oriented end-user programming

Tomas Petricek and Jonathan Edwards
tomas@tomasp.net | @tomasp.net

Figure 1. Program is a sequence of edits that can be replied to construct a document.

Researchers have envisioned many programming experiences that make programming easier. In programming by demonstration, we create programs by showing what we want to achieve using specific values, while concrete programming is built around the idea of copying and modifying parts of programs. We also need support for collaborative editing, on-demand recomputation when parts of a program change, and automatic updating of code when a relevant schema changes.

Building a programming system that supports multiple such features is hard. Denicek is a computational substrate that represents programs as sequences of interactions that edit a document. It provides primitive operations for applying edits, merging edits, and detecting conflicts. As it turns out, many of the above programming experiences can be easily built on top of this simple structure.

This interactiv essay demonstrates a basic web-based programming system based on Denicek. It walks you through the solutions to a couple of standard challenge problems, like building a counter app, TODO list, and managing a conference. The demos replay actions using the real Denicek implementation, so you can also explore on your own!

Programming experiences

There are many programming experiences that go beyond just writing code. In programming by demonstration (Smith, 1975) programs are created by demonstrating the proces using concrete values, suport for collaborative editing (Kleppmann et al., 2019) is vital for working togther, incremental recomputation (Teitelbaum et al., 1981) gives fast feedback by only reevaluating parts of programs, schema change control (Edwards et al., 2025) can automatically update program when the structure of data changes, end-user debugging (Ko et al., 2009) lets us understand where values came from, and concrete programming (Edwards, 2005) makes it possibly to create abstractions using managed copy and paste.

The Denicek substrate

The problem is that supporting these programming experiences in a programming system based on underlying textual substrate is hard. Denicek is a computational substrate that makes it possible to build programming systems that support the above.

The key idea in Denicek is to represent programs as a series of edits that transform a document. Data and code are nodes in the document. Programming or editing data results in document edits. Entering data into textboxes in the document also yields document edits.

The Denicek substrate provides three basic operations on edit histories. It can replay the history to construct the current version of a document. It can merge two edit histories that have diverged and it can detect conflicts between branches. It turns out that many of the above programming experiences can be expressed in terms of these three primitive operations.
