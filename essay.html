Denicek: Computational substrate for document-oriented end-user programming

Tomas Petricek and Jonathan Edwards
tomas@tomasp.net | @tomasp.net

Figure 1. Program is a sequence of edits that can be replied to construct a document.

Researchers have envisioned many programming experiences that make programming easier. In programming by demonstration, we create programs by showing what we want to achieve using specific values, while concrete programming is built around the idea of copying and modifying parts of programs. We also need support for collaborative editing, on-demand recomputation when parts of a program change, and automatic updating of code when a relevant schema changes.

Building a programming system that supports multiple such features is hard. Denicek is a computational substrate that represents programs as sequences of interactions that edit a document. It provides primitive operations for applying edits, merging edits, and detecting conflicts. As it turns out, many of the above programming experiences can be easily built on top of this simple structure.

This interactiv essay demonstrates a basic web-based programming system based on Denicek. It walks you through the solutions to a couple of standard challenge problems, like building a counter app, TODO list, and managing a conference. The demos replay actions using the real Denicek implementation, so you can also explore on your own!

Chapter 1: Normalizing hello world messages

Step 1: Creating list of messages

In the first demo, we create a list of messages with MixEd caPiTAliZaTIoN and normalize them by recording and replaying a simple transformation. To get started, click the button to replay the first part of the demo. You can also use and to skip to the previous or next step.

In the first step, we enter :hellos=<ul> to add a new record with the <ul> tag as a field hellos of the document. We then add multiple <li> records as children and set their field greeting to a greeting in various languages.

The demo controls the actual Denicek, so you can also explore on your own! Use arrows to move around the document, type ? to open the command palette. You can use Alt+H to toggle history view and Alt+U to view document source. If you play around, make sure to use to reset the demo!

Step 2: Fix capitalization of one item

To fix the capitalization, we will copy the greeting and transform the two copies. We move to the first item in the record (en field) and copy the value of its greeting field (the text) using Alt+C. We then add the copy as another field named rest using :rest=!v. Note that Denicek treats this as a special copy edit with source (greeting) and target (rest).

To fix the capitalization, we apply a number of built-in primitive transformations to the two text fields. @take-first and @upper turn the first field into capitalized first letter; @skip-first and @lower turn the second field into lower-case rest of the greeting.

Step 3: Save interactions for replay

Denicek represents documents as sequences of edits. You can see the edits in the History table on the right. Use Alt+H to display or hide it. In the table, you can:

  • hover over the edit name to see more details,
  • hover over a selector to highlight the affected node,
  • click a hash to go back to a previous document state,
  • click edit number to select or unselect it.

Now, we select the last 6 edits (watch the history table!) that create the rest field, copy the greeting and transform the fields and save them as saved interactions using !s normalize.

Note that edits are saved in the document! You'll see a number of append edits that add nodes to /saved-interactions/normalize and you can see those if you view the source of the document. After the demo finishes playing, hit the up arrow until you reach the root and then use Alt+U.

Step 4: Replaying normalization edits

To clean up the capitalization of the remaining fields, we now go over the individual record fields and apply our saved normalize operation to each of the fields.

We need to use @normalize! that lets us specify that the currently selected document node should be used in place of /hellos/en in the saved interactions. The @normalize operation just replays the edits on the original targets.

As we will see in Chapter 3, there is a better way of doing this. Denicek supports lists, which let us apply operations on all elements of the list at the same time. The <ul> element in this demo was Denicek's record. We could have created it as Denicek's list, but that would not illustrate replaying of edits!

After replaying the demo, you can undo the edits using Alt+Z and try on your own! Move around using keybaord arrows, and type @ to see the saved operation and apply it!

Step 5: Next steps

This demo was the "Hello world" of Denicek. You saw the basic idea: representing documents as sequences of edit operations. You saw that Denicek can save and replay past interactions to support programming by demonstration. The remaining chapters show that Denicek can do much more than that:

  • Chapter 2: Todo list is a bigger example that creates a user interface using programming by demonstration.
  • Chapter 3: Conference list shows collaborative editing and handling of schema change.
  • Chapter 4: Conference budget adds formulas with incremental recomputation and debugging.
  • Chapter 5: Counter combines formulas with programming by demonstration.
  • Chapter 6: Traffic accidents shows how concrete programming enables tracking of code copies.
  • Chapter 7: Under the cover provides some more background on how Denicek works.
